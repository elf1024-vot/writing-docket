System Requirements

02

Getting Started - where to start?

You can fill everything by hand in the editors, or hand Claude Code a plain list and let it scaffold the rows for you. Either way, build your world in dependency order so the pieces that reference each other resolve:

  1. Characters first: nothing depends on them, and everything else points at them.
  2. Locations next: also independent.
  3. Groups, which can name a leader (a character), so add them after characters.
  4. Book & chapters last: chapters point at a POV character and a location.

The fastest way to scaffold each is the helpers in Claude Code. You give plain, natural values: just names. The database already defines the keys and the links between tables, so Claude generates the ids and wires up the connections for you (a group's leader, a chapter's POV character). It does not write the descriptive content: backgrounds, descriptions, summaries are left blank for you to author in the editors:

To add…Run in Claude CodeYou just say…
Characters/novel-add-charactersNames (optionally a role)
Locations/novel-add-locationsPlace names
Groups / factions/novel-add-groupGroup names, optionally "led by [character]"
A book + blank chapters/novel-add-bookBook name + how many chapters (optionally "Ch 1 is [character] at [place]")

Each helper shows the rows it intends to create and asks for a yes first; if you name a leader or POV who isn't in the cast yet, it tells you so you can add them first; and re-running is safe (existing rows are skipped). The structure and the links are built for you; the writing is yours.

Prefer a spreadsheet? Your project folder has an intake\ folder with four fill-in files: characters.csv, locations.csv, groups.csv, chapters.csv. Fill them in locally in Excel or LibreOffice (or Google Sheets if you prefer), delete the [EXAMPLE] row, save back as CSV, then run /novel-import in Claude Code. It builds and links everything the same way. See intake\README.txt for details.

Recommended: stay offline. The CSV files never leave your computer, so your story bible stays private and works with no internet. We recommend the local spreadsheet route over any cloud sync.

What is the MCP?

The MCP (Model Context Protocol) server is the relay that lets Claude read and write your project database directly through well-scoped tools, instead of you copy-pasting. It connects to the same PostgreSQL the web UI uses, so edits made by Claude and edits made in the editors stay in sync.

What are example rows?

A fresh build seeds a couple of placeholder rows (an example character, location, and chapter) so the editors and dashboard render something on first run. Delete them once you start entering real content; they are only scaffolding.

What are Gates?

Gates (G0-G8, plus any custom ones you add) are quality checks the QR assessment applies to a chapter. In plain English:

GateMeaning
G0Content safety (absolute): runs first; a failure forces a 0.0 score.
G1Point of view is held cleanly, no unearned head-hops.
G2Show, don't tell: emotion is demonstrated, not announced.
G3Every scene has a purpose; something changes.
G4Character voice is distinct and consistent.
G5Dialogue is purposeful: each line does more than one job.
G6Pacing matches the scene's emotional register.
G7Sensory grounding: at least two senses engaged per scene.
G8No clichés, AI-isms, or hollow intensifiers.

Each gate carries a fixed penalty (G0 is the exception; it's absolute). Your own custom gates from the plan doc are added below these.

07

What is a QR score?

QR (Quality Rating) is a 0-10 score for a chapter's craft. On the dashboard and tracker it is color-coded: green ≥ 8.5, yellow 7-8.4, red < 7. It is read-only in the editors; it is set by the QR assessment, not typed by hand.

Scoring is gated. First a Content Safety check and a Scene Rendering check must pass. Then the Canary metrics act as pass/fail pre-conditions: a script (scripts/canary.py) measures passive voice, emotion tells, weak adverbs, sentence variety, complex paragraphs, readability, and glue index against your genre's thresholds. If any metric fails, the chapter is returned for rework and no number is calculated until you fix it (you can override a specific metric with a documented reason). Only once everything passes are the craft gates (G1-G8) scored, each with a fixed penalty: QR = 10.0 minus the penalties. Because the metrics are computed, the same chapter always evaluates the same way; a change in the numbers is a real change in the prose, not the reviewer's mood.

How do I add a custom gate?

Edit prompts/Standards.md and add your rule. The QR and Polish workflows read that file, so a new standard there becomes part of the gate checks automatically.

09

How do I back up?

Back up often. Run bat-files\backup.bat to dump the database into backups\. Then copy backups\ to a USB drive or external disk. Docker volumes live on one disk; if that disk dies, an on-disk backup dies with it.
10

What does profile_size mean?

profile_size is a read-only measure of how fully a character is described (length of their combined profile fields). The dashboard shows it as an inline bar so you can spot thin characters at a glance.

11

Can I run multiple projects?

Yes. Each project is its own folder with its own Docker stack, database, UI port, and MCP slug. Give each a distinct {{UI_PORT}} and MCP name so they do not collide.

12

How do I add a genre extension later?

Run /system-extend to layer a genre extension onto an existing project (extra fields, prompts, or standards) without rebuilding.

13

How do I uninstall?

Run /system-uninstall. Back up first; uninstalling removes the stack and its data.

14

Protect against hardware failure

Keep a USB copy of backups\. The single most important habit: an off-machine copy of your backups means a dead disk costs you hardware, not your manuscript.

How does search work?

Search is built in and on by default; open it from the Search tab. It works two ways, both fully offline - nothing about your story ever leaves your machine:

  • Text (full-text) search is always available. PostgreSQL indexes the title and body of every character, location, group, chapter, note, and chapter prose file the moment it is written, so keyword search just works - even if the embedding service is off.
  • Semantic search finds content by meaning, not just exact words. A local ollama container running the nomic-embed-text model turns your query into a 768-number vector, and PostgreSQL (with the pgvector extension) returns the closest matches. The model is pulled once from the internet (~300MB) during setup; after that it runs entirely on your machine.

The semantic index is built by scripts/embed.py (run automatically at build, and re-runnable any time you add content). If the Ollama container is not running, semantic search degrades gracefully: the Search page shows a banner and falls back to text search, which always works. No part of search requires the internet at runtime.
